# 05. 核心页面信息架构

## 1. 目标

定义 MVP-0 的产品页面、信息层级、用户路径和关键交互，确保界面承载结构化圆桌，而不是退化为聊天应用。

## 2. 顶层导航

```text
问题工作台
Agent 库
圆桌记录
模板中心
设置
```

MVP-0 推荐默认进入“问题工作台”。

## 3. 页面一：问题工作台

### 3.1 页面目标

帮助用户把复杂问题输入为可分析、可调度、可审议的任务。

### 3.2 核心模块

- 问题输入区。
- 诉求输入区。
- 背景材料区。
- 目标与输出格式。
- 权限与保存策略。
- 任务画像预览。
- Agent 推荐预览。

### 3.3 用户动作

- 输入问题。
- 输入诉求。
- 粘贴背景。
- 上传或选择材料。
- 选择用途：方案、评审、产品规划、技术路线、客户话术、研究分析。
- 选择输出：报告、行动清单、PPT 大纲、PRD、风险评审。
- 设置是否保存。
- 设置是否允许联网。
- 点击“生成任务画像”。

### 3.4 体验要求

- 不要让用户一开始手选 Agent。
- 问题输入后应先显示任务画像，而不是直接开聊。
- 任务画像必须可编辑。
- 系统必须提示当前缺少哪些信息。

## 4. 页面二：Agent 推荐与阵容确认

### 4.1 页面目标

让用户理解系统为什么推荐这些 Agent，并允许调整。

### 4.2 核心模块

- 推荐阵容。
- 角色分工。
- 覆盖矩阵。
- 排除 Agent 列表。
- 缺失视角提示。
- 权限标签。
- Agent 类型标签：基础公共、个人、共享。

### 4.3 展示方式

推荐阵容不宜只用头像卡片，应使用“能力覆盖矩阵”：

```text
问题维度        推荐 Agent                 覆盖状态
产品定位        Product Manager           已覆盖
体验路径        UX Architect              已覆盖
界面表达        UI Designer               已覆盖
工作流编排      Workflow Architect         已覆盖
多 Agent 治理   Multi-Agent Architect      已覆盖
技术架构        Software Architect         已覆盖
安全边界        Security Architect         已覆盖
身份信任        Identity Trust Architect   已覆盖
隐私保护        Data Privacy Officer       已覆盖
```

### 4.4 用户动作

- 确认阵容。
- 替换 Agent。
- 添加质疑者。
- 降级某 Agent 为观察员。
- 查看 Agent 详情。
- 调整材料访问权限。

## 5. 页面三：圆桌现场

### 5.1 页面目标

让用户看到讨论正在按结构推进，并能理解每个阶段产出了什么。

### 5.2 页面布局

推荐三栏结构：

```text
左栏：阶段与 Agent
中栏：当前讨论板
右栏：争议、证据、待确认
```

### 5.3 左栏

包含：

- 当前阶段。
- 参与 Agent。
- Agent 所属类型与归属人。
- 每个 Agent 状态：等待、发言中、已完成、失败、被阻止。
- 权限标签。
- 轮次进度。

### 5.4 中栏

按阶段展示：

- 首轮观点卡。
- 质疑卡。
- 回应卡。
- 修正卡。
- 共识卡。

每张卡必须显示：

- 来源 Agent。
- 观点摘要。
- 信心等级。
- 相关证据。
- 是否被质疑。
- 是否已修正。

### 5.5 右栏

包含：

- 当前争议点。
- 高风险提醒。
- 证据缺口。
- 用户可补充材料入口。
- “要求更强反驳”按钮。
- “提前综合”按钮。

### 5.6 体验要求

- 不要用普通聊天气泡作为主体。
- 用户能一眼看出讨论有没有进入质疑阶段。
- 用户能一眼看出哪些观点还没被挑战。
- 用户能中途暂停，不丢失结果。

## 6. 页面四：争议地图

### 6.1 页面目标

把圆桌讨论中的分歧、质疑和修正可视化。

### 6.2 核心模块

- 争议点列表。
- 观点与质疑关系图。
- 已解决和未解决分歧。
- 影响等级。
- 处理建议。

### 6.3 争议状态

- `open`：未回应。
- `responded`：已回应。
- `revised`：已导致修正。
- `resolved`：已解决。
- `deferred`：待证据确认。

### 6.4 体验要求

- 争议地图要比聊天记录更重要。
- 用户能按严重程度过滤。
- 用户能点击争议跳回原始观点。

## 7. 页面五：证据台账

### 7.1 页面目标

让用户知道哪些结论有依据，哪些仍需确认。

### 7.2 核心模块

- 输入材料列表。
- Agent 使用记录。
- 证据链接。
- 证据缺口。
- 待联网核验事项。
- 敏感信息提示。

### 7.3 证据状态

- 已由用户输入支撑。
- 已由上传材料支撑。
- 仅为 Agent 推理。
- 需要联网核验。
- 与已有材料冲突。
- 不足以支持结论。

## 8. 页面六：输出中心

### 8.1 页面目标

生成可直接使用的最终产物，并保留追溯入口。

### 8.2 输出类型

MVP-0 支持：

- Markdown 评审报告。
- 行动清单。
- 证据缺口清单。
- Agent 讨论摘要。
- 结构化 JSON 包。

后续支持：

- PPT 大纲。
- PRD。
- 技术方案。
- Word 文档。
- 可分享网页。

### 8.3 报告结构

默认报告包含：

- 一句话结论。
- 任务画像。
- 参与 Agent。
- 核心建议。
- 主要争议。
- 已修正观点。
- 致命缺陷。
- 高危缺陷。
- 待确认前提。
- 下一步行动。
- 事件索引。

## 9. 页面七：Agent 库

### 9.1 页面目标

管理现有员工，并按照设计模式新增员工。

### 9.2 核心模块

- Agent 列表。
- Agent 诉求入口。
- 来源筛选。
- 类型筛选：基础公共、个人、共享。
- 信任状态筛选。
- 能力标签筛选。
- Agent 详情。
- 版本历史。
- 测试记录。
- 新增 Agent。

### 9.3 Agent 详情页

必须展示：

- 名称。
- 来源。
- 类型。
- 归属人。
- 当前版本。
- 信任状态。
- 能力标签。
- 适用场景。
- 不适用场景。
- 权限需求。
- 最近使用表现。
- 测试用例。
- 是否可共享。

### 9.4 新增 Agent 页面

流程：

1. 选择创建方式：从诉求生成、从模板创建、从导入文件创建。
2. 填写或确认用户诉求。
3. 系统先尝试匹配已有 Agent。
4. 若匹配不足，生成个人 Agent 草稿。
5. 填写 Agent 身份。
6. 填写能力和边界。
7. 填写工作方式。
8. 设置权限需求。
9. 添加测试问题。
10. 试运行。
11. 保留为个人 Agent、申请共享或进入推荐池。

## 10. 页面八：模板中心

### 10.1 页面目标

把一次有效圆桌沉淀成可复用资产。

### 10.2 模板类型

- 任务画像模板。
- Agent 组合模板。
- 圆桌轮次模板。
- 质疑清单模板。
- 报告结构模板。

### 10.3 体验要求

- 模板来源必须可追溯。
- 模板不能自动污染默认流程。
- 用户能从历史 Session 生成模板。

## 11. 页面九：设置

MVP-0 设置项：

- Agent 源路径。
- 本地保存路径。
- 默认材料保存策略。
- 默认联网策略。
- 默认圆桌轮次。
- 导出格式。

## 12. MVP-0 页面范围

必须做：

- 问题工作台。
- Agent 推荐与阵容确认。
- 圆桌现场。
- 输出中心。
- Agent 库。

可以简化：

- 争议地图作为圆桌现场右栏。
- 证据台账作为输出中心子页。
- 模板中心只做保存与列表。
- Agent 诉求驱动创建必须保留。

暂不做：

- 团队协作。
- Agent 市场。
- 复杂仪表盘。
- 商业化计费。

## 13. 体验验收

- 用户不用阅读说明也知道从哪里开始。
- 用户能在圆桌现场看出讨论不是普通聊天。
- 用户能理解为什么选这些 Agent。
- 用户能看出结论是否经过质疑。
- 用户能从最终报告回到争议和证据。
- 用户能新增 Agent，但不会误以为新增后立即可信。
