# 06. MVP-0 验收标准

## 1. MVP-0 定义

MVP-0 是一个本地优先的 Agent 圆桌工作台，验证核心体验是否成立：

用户输入一个复杂问题，系统能分析任务、推荐 Agent、组织结构化圆桌、展示争议和证据缺口，并输出可追溯报告；用户还能按照现有 Agent 设计模式新增员工，并通过测试后进入可推荐状态。

MVP-0 不追求商业化，不追求团队协作，不追求华丽动效。MVP-0 追求“真的好用，真的能帮助用户想清楚复杂问题”。

## 2. 必须完成的用户故事

### US-01：创建圆桌任务

作为用户，我可以输入一个复杂问题、补充背景材料、选择输出目标，让系统生成任务画像。

验收标准：

- 用户可以输入问题和背景。
- 用户可以选择或填写输出目标。
- 系统生成任务画像。
- 任务画像至少包含任务类型、关键约束、所需专业视角、证据需求和风险等级。
- 用户可以编辑任务画像。

### US-02：自动推荐 Agent

作为用户，我不需要手工挑选员工，系统可以根据任务画像推荐合适 Agent。

验收标准：

- 系统能从本地 Agent 源读取候选。
- 系统能区分基础公共、个人和共享三类 Agent。
- 推荐阵容至少包含 3 个 Agent。
- 每个推荐 Agent 都有推荐理由。
- 系统展示被排除候选及排除原因。
- 用户可以替换、增加或移除 Agent。
- 系统显示当前阵容缺失的专业视角。
- 系统能先匹配已有 Agent，再决定是否生成新的个人 Agent。

### US-03：启动结构化圆桌

作为用户，我可以启动圆桌，让 Agent 按阶段输出观点、质疑、回应、修正和共识。

验收标准：

- 圆桌至少完成首轮观点。
- 圆桌至少完成一轮交叉质疑。
- 每条高优先级质疑都有回应。
- 至少有一个观点发生修正，或系统明确说明没有修正的原因。
- 系统区分共识、分歧和待确认前提。

### US-04：查看讨论过程

作为用户，我可以看到圆桌讨论过程，而不是只看到最终总结。

验收标准：

- 页面显示当前阶段。
- 页面显示每个 Agent 的状态。
- 页面显示观点卡、质疑卡、回应卡、修正卡。
- 页面显示争议点。
- 页面显示证据缺口。
- 用户可以要求更强反驳。
- 用户可以暂停并继续。

### US-05：生成可追溯报告

作为用户，我可以得到一份可直接使用的最终报告，并能追溯结论来源。

验收标准：

- 报告包含一句话结论。
- 报告包含参与 Agent 和版本。
- 报告包含核心建议。
- 报告包含主要争议。
- 报告包含已修正观点。
- 报告包含致命缺陷和高危缺陷。
- 报告包含证据缺口。
- 报告包含待确认前提。
- 报告包含下一步行动。
- 报告中的关键结论能追溯到结构化事件。

### US-06：新增或生成 Agent

作为用户，我可以通过导入、表单或诉求驱动的方式新增一个员工，并让它按照现有 Agent 设计模式进入试运行。

验收标准：

- 用户可以从 TOML、Markdown 或表单新增 Agent。
- 用户可以直接输入诉求，让系统先匹配已有 Agent，必要时生成个人 Agent 草稿。
- 系统记录来源和版本。
- 系统明确该 Agent 是基础公共、个人还是共享。
- 系统解析能力标签。
- 系统要求填写适用边界和不适用场景。
- 新增 Agent 默认是试运行状态，且个人 Agent 默认仅归属创建者。
- 新增 Agent 不能直接进入默认推荐池。
- 用户可以为 Agent 添加测试问题。
- 试运行通过后，用户可以批准进入默认推荐池。
- 个人 Agent 可以被用户授权转为共享 Agent。
- MVP-0 至少保留共享池状态和发布记录；真实“其他用户可用”作为云端部门版验收项。

### US-07：保存为模板

作为用户，我可以把一次有效圆桌保存为模板，供以后类似问题复用。

验收标准：

- 用户可以保存任务画像模板。
- 用户可以保存 Agent 组合模板。
- 用户可以保存质疑清单。
- 用户可以保存输出结构。
- 保存模板前系统显示模板来源。
- 模板不会默认覆盖全局流程。

## 3. 功能性要求

### FR-01 任务画像

系统必须从用户问题中生成结构化任务画像。

### FR-02 Agent 读取

系统必须能读取本地 `.codex/agents/*.toml`。

### FR-03 源文件关联

当 Agent 来自 `agency-agents`，系统应尽量关联其源 Markdown。

### FR-04 Agent 推荐解释

系统必须解释 Agent 推荐和排除原因。

### FR-05 阵容确认

用户必须能在圆桌开始前确认或调整阵容。

### FR-06 结构化讨论事件

系统必须保存观点、质疑、回应、修正、共识、分歧、证据缺口等结构化事件。

### FR-07 状态机可见

系统必须显示圆桌当前阶段。

### FR-08 质疑门禁

没有质疑的圆桌不能标记为完整。

### FR-09 证据缺口

最终报告必须显示证据缺口。

### FR-10 Agent 版本追踪

最终报告必须显示使用的 Agent 版本。

### FR-11 新增 Agent 试运行

新增 Agent 默认不能进入默认推荐池。

### FR-12 Agent 三类资产

系统必须区分基础公共、个人、共享三类 Agent。

### FR-13 诉求驱动生成

系统必须支持用户直接提出诉求，并先匹配已有 Agent；若覆盖不足，再生成个人 Agent 草稿。

### FR-14 权限默认关闭

文件访问、联网、工具调用默认关闭或受限。

### FR-15 用户保存确认

上传材料和完整 Session 默认不长期保存，除非用户主动确认。

### FR-16 导出

系统必须支持 Markdown 报告导出和结构化 JSON 导出。

## 4. 非功能性要求

### NFR-01 可理解性

第一次使用的用户应能在 3 分钟内理解主流程。

### NFR-02 完成时长

默认圆桌任务应在 10 分钟内完成一次可用输出。

### NFR-03 失败可恢复

单个 Agent 失败不应导致整个任务失败。

### NFR-04 可追溯

最终报告中的关键结论必须能追溯到事件链。

### NFR-05 隐私默认保护

用户材料默认不进入长期资产库。

### NFR-06 体验优先

任何功能如果会把产品退化为聊天流，应推迟或重做。

## 5. 首版不做

- 不做商业化。
- 不做 Agent 市场。
- 不做多人在线协作。
- 不做复杂组织权限。
- 不做跨端同步。
- 不做真实部门云端共享。
- 不做自动发布到外部平台。
- 不做全自动联网事实核验。
- 不做无限自由 mesh 讨论。
- 不做通用聊天入口。

## 6. 质量门禁

MVP-0 发布前必须通过：

- 3 个不同类型问题的端到端圆桌测试。
- 至少 20 个本地 Agent 的读取和解析测试。
- 至少 3 个新增 Agent 的导入、试运行和批准测试。
- 至少 5 次 Agent 推荐解释检查。
- 至少 5 份最终报告追溯检查。
- 至少 3 次异常流程测试：Agent 失败、证据不足、权限阻止。

## 7. 体验验收问题

每次验收必须回答：

1. 用户是否知道当前进行到哪一步？
2. 用户是否知道为什么选这些 Agent？
3. 用户是否看到了真实质疑？
4. 用户是否看到了观点修正？
5. 用户是否知道哪些结论有证据、哪些没有？
6. 用户是否能直接使用最终报告？
7. 用户是否能把这次圆桌沉淀为模板？
8. 新增 Agent 是否被默认隔离在试运行状态？

只要有两个以上答案是否定，MVP-0 不合格。

## 8. 样例验收任务

### 任务 A：产品规划

输入：把基于 agency-agents 和 multiagent 的工作流做成可视化应用。

期望：

- 推荐产品、UX、UI、工作流、多 Agent、架构、安全、隐私、信任角色。
- 生成产品体验方案。
- 明确不能做成聊天应用。

### 任务 B：技术选型

输入：为这个产品设计本地优先架构。

期望：

- 推荐软件架构、多 Agent、数据、安全、隐私角色。
- 输出模块化单体加 Worker 的方案。
- 明确数据模型和权限边界。

### 任务 C：新增 Agent

输入：新增一个“医疗合规评审 Agent”。

期望：

- 系统要求填写能力、边界、权限、测试用例。
- 或者先从诉求生成个人 Agent 草稿。
- 默认标记试运行。
- 不进入默认推荐池。
- 如用户授权，可后续转为共享 Agent。

### 任务 D：部门云端版本预留

输入：未来希望部署到云上给部门同事使用。

期望：

- 当前 MVP-0 不扩大为多人系统。
- 数据模型保留用户、归属、可见性和共享状态。
- 后续版本补组织空间、权限、审计、共享审核和数据隔离。

## 9. 发布判定

MVP-0 可以发布给自己使用的最低标准：

- 端到端流程可跑通。
- 报告质量明显优于单 Agent 回答。
- 讨论过程可观察。
- 结论可追溯。
- 新增 Agent 不污染推荐池。

如果只能展示多个 Agent 依次发言，而不能展示争议、修正、证据和采纳理由，则不得发布。
