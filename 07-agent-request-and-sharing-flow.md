# 07. 诉求驱动的 Agent 匹配、生成与共享

## 1. 目标

定义用户提出“我需要什么样的员工”之后，系统如何先匹配已有 Agent，不足时再按生成规则创建个人 Agent，并允许个人 Agent 经过授权后转为共享 Agent。

## 2. Agent 三类资产

### 2.1 基础公共 Agent

基础公共 Agent 是系统自带 Agent。

规则：

- 所有用户可见、可用。
- 由系统或平台维护。
- 默认作为首选基础池。
- 普通用户不能直接覆盖原始版本。
- 可被复制为个人 Agent 后再修改。

适用：

- 通用高频任务。
- 标准专业角色。
- 产品、UX、UI、架构、安全、隐私、文档等基础能力。

### 2.2 个人 Agent

个人 Agent 是归属于单个用户的 Agent。

规则：

- 默认仅创建者可见、可用。
- 可以由用户诉求生成。
- 可以由基础公共 Agent 或共享 Agent 复制后修改。
- 默认处于个人库，不进入共享池。
- 默认不参与其他用户的推荐。

适用：

- 个人工作习惯。
- 特定行业角色。
- 临时项目角色。
- 用户私有方法论。
- 不适合公开的部门或客户上下文。

### 2.3 共享 Agent

共享 Agent 是被发布给所有用户使用的 Agent。

规则：

- 所有用户可见、可用。
- 可以由个人 Agent 转化而来。
- 必须保留来源与版本链。
- 必须经过共享发布检查。
- 可以被其他用户复制为个人 Agent 后再修改。
- 被撤回后不再进入默认推荐池。

适用：

- 经验证的高质量专业角色。
- 可复用的行业角色。
- 部门内可共用的经验沉淀。
- 未来云端部门空间中的共享资产。

## 3. 诉求驱动流程

### 3.1 用户提出诉求

用户可以不用知道 Agent 名称，直接描述：

- 我现在要解决什么问题。
- 我需要什么样的专家。
- 我希望这个 Agent 输出什么。
- 我希望它怎么参与圆桌。
- 我希望它关注哪些风险。

示例：

```text
我需要一个专门评审医疗合规风险的 Agent，能检查健康数据、用户授权、宣传用语和隐私边界。
```

### 3.2 系统先匹配已有 Agent

系统必须先在用户可见范围内检索：

- 基础公共 Agent。
- 用户自己的个人 Agent。
- 已发布共享 Agent。

输出必须包含：

- 已匹配 Agent。
- 覆盖的能力。
- 未覆盖的能力。
- 是否建议创建新个人 Agent。
- 为什么不直接复用某些候选 Agent。

### 3.3 不足时生成个人 Agent

当现有 Agent 无法覆盖诉求，系统生成个人 Agent 草稿。

草稿必须包含：

- 名称。
- 一句话职责。
- 身份与工作方式。
- 能力标签。
- 适用场景。
- 不适用场景。
- 输出结构。
- 禁止行为。
- 权限需求。
- 测试问题。
- 默认归属人。
- 默认可见性。

默认状态：

- `agent_class = personal_private`
- `visibility_scope = owner_only`
- `trust_status = trial_only`
- `default_pool_eligible = false`

### 3.4 个人 Agent 试运行

个人 Agent 必须先试运行。

试运行检查：

- 是否解决了原始诉求。
- 是否有清晰边界。
- 是否产生实质贡献。
- 是否和已有 Agent 重复过高。
- 是否存在越权、隐私或安全风险。

### 3.5 个人 Agent 进入日常使用

试运行通过后，用户可以：

- 仅在当前任务使用。
- 保存到个人 Agent 库。
- 加入个人推荐池。
- 继续编辑。
- 发起共享。

## 4. 共享发布流程

### 4.1 发起共享

用户可以将个人 Agent 发起共享。

发起共享前必须展示：

- 共享后谁可以使用。
- 共享内容包含哪些指令和元数据。
- 是否包含个人、客户、部门或敏感信息。
- 是否允许他人复制后修改。
- 原始个人 Agent 是否继续保留。

默认规则：

- 原始个人 Agent 继续保留。
- 共享 Agent 是一个新发布版本。
- 共享 Agent 不反向覆盖个人 Agent。

### 4.2 共享检查

共享前必须检查：

- 是否包含敏感信息。
- 是否包含个人身份、客户名称或内部资料。
- 是否存在越权权限需求。
- 是否存在不稳定或过度承诺的能力声明。
- 是否与已有共享 Agent 重复过高。

### 4.3 发布为共享 Agent

共享发布后生成：

- `shared_agent_id`
- `published_from_agent_id`
- `publisher_user_id`
- `published_version_id`
- `published_at`
- `sharing_scope`
- `reuse_policy`

共享范围：

- MVP-0：本地当前用户环境内的共享概念先保留为数据模型。
- 后续云端：可扩展为部门空间、团队空间、全组织空间。
- 部门云端版：必须补齐账号、组织、权限、审计和数据隔离后，才能让部门同事实际使用共享 Agent。

## 5. 推荐与排序

### 5.1 默认搜索顺序

建议默认顺序：

1. 基础公共 Agent。
2. 用户个人 Agent。
3. 共享 Agent。

排序不能只按类别，必须综合：

- 能力匹配度。
- 适用边界。
- 信任状态。
- 历史表现。
- 最近使用情况。
- 当前任务覆盖缺口。

### 5.2 推荐解释

系统必须解释：

- 这个 Agent 是基础公共、个人还是共享。
- 为什么推荐它。
- 它覆盖哪部分诉求。
- 它是否经过验证。
- 它能看到哪些材料。

## 6. 云端部门版本预留

未来如果部署到云上给部门同事使用，需要补齐：

- 用户账号。
- 部门空间。
- 角色权限。
- 共享审核。
- 使用审计。
- 数据隔离。
- Agent 版本发布记录。
- 共享 Agent 撤回机制。
- 部门知识资产沉淀。

云端版本不能简单把本地个人库同步上去。必须先把个人、部门、共享边界拆清楚。

## 7. 验收标准

- 用户只描述诉求，系统就能先推荐已有 Agent。
- 若已有 Agent 不足，系统能生成个人 Agent 草稿。
- 个人 Agent 默认只归属创建者。
- 用户能把个人 Agent 发起共享。
- 共享后，其他用户能在自己的任务里使用该 Agent。
- 系统始终能区分基础公共、个人、共享三类 Agent。
- 任何 Agent 的来源和版本都能追溯。
- 云端部门版的需求被记录为未来版本，不影响 MVP-0 的本地优先范围。
