# Agent Roundtable Studio 质量验收

本产品不再以“流程跑通”作为完成标准。每次发布至少验证以下三类真实问题，确认系统能自动识别任务、组建合理 Agent、产生质疑链路并输出可执行报告。

## 1. 教育场景

输入：

```text
如何将 AI 赋能到英语教学当中，尤其是对于硬件条件比较弱的初中学校？
```

必须出现：

- 任务画像识别为教育/教学场景。
- 必要视角包含教学设计、学习评估、学校落地、数据隐私。
- Agent 阵容包含教学设计、学习评估或学校落地相关 Agent。
- 质疑必须围绕低硬件条件、教师工作流、学习效果证据，而不是泛泛讲 AI。
- 报告必须给出可执行试点路径和缺失信息。

## 2. 医疗健康上线评审

输入：

```text
我要准备一个医疗健康产品上线评审，系统必须自动组建圆桌并补齐合规、行业专家和财务审计视角。
```

必须出现：

- 任务画像识别为医疗健康高风险场景。
- 必要视角包含合规审查、数据隐私、医疗行业专家、财务审计。
- 质疑必须触及健康数据、用户授权、宣传边界、成本收益证据。
- 报告不得把医疗或合规结论写成最终法律/医疗意见。

## 3. 部门工具上线规划

输入：

```text
我们准备把这个圆桌工具放到云上给部门同事使用，如何设计上线方案？
```

必须出现：

- 任务画像识别为技术产品上线或部门协作工具场景。
- 必要视角包含产品价值、技术架构、AI 编排、安全架构、数据隐私、现实校验。
- 质疑必须触及权限、审计、密钥、成本、真实使用质量，而不是只讲部署。
- 报告必须区分首期必做、后续增强、上线前阻塞项。

## 通过标准

- 每类问题必须生成 `position`、`challenge`、`response`、`revision`、`consensus`、`evidence_gap` 事件。
- 第三步“系统组建圆桌”必须展示运行证据，包括 schema、runtime、模型或降级状态、耗时。
- 如果 `runtime_mode` 不是 `openai`，只能判定为开发联调通过，不能判定为生产质量通过。
- 报告必须包含明确行动建议、证据缺口和剩余风险。
