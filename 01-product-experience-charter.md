# 01. 产品体验总纲

## 1. 产品一句话

Agent Roundtable Studio 是一个面向复杂问题的结构化审议工作台：用户输入问题后，系统自动分析任务、匹配专业 Agent、组织有质疑和修正的圆桌讨论，并输出可追溯、可复用的高质量结论。

## 2. 产品不是

- 不是多 Agent 聊天室。
- 不是头像轮流发言的演示页面。
- 不是 Prompt 管理器。
- 不是只用于程序员的代码 Agent 面板。
- 不是 Agent 市场。
- 不是把 `agency-agents` 的文件目录换成网页列表。

## 3. 产品是

- 一个“问题工作台”：帮助用户把模糊问题变成可审议任务。
- 一个“专家调度台”：根据问题选择合适 Agent，而不是让用户手工挑人。
- 一个“圆桌审议现场”：让观点、质疑、回应、修正和共识有结构地发生。
- 一个“争议地图”：把讨论从聊天流变成可理解的关键分歧。
- 一个“证据台账”：标记证据来源、证据缺口、未验证前提和采纳依据。
- 一个“结论生成器”：输出经过专业讨论、质疑和优化的方案。
- 一个“诉求驱动入口”：用户先提需求，系统先匹配已有 Agent，不足时再生成个人 Agent。
- 一个“三层 Agent 资产系统”：基础公共 Agent、个人 Agent、共享 Agent。
- 一个“资产复用中心”：把高质量任务画像、Agent 组合、质疑清单和输出格式沉淀为模板。

## 4. 目标用户

### 4.1 主要用户

- 需要高质量方案的个人工作者。
- 产品经理、咨询顾问、项目负责人、技术负责人。
- 需要跨专业视角做判断的业务人员。
- 经常写方案、评审材料、产品规划、技术路线和客户沟通材料的人。

### 4.2 非首版用户

- 需要多人在线协作的组织团队。
- 希望买卖 Agent 的创作者。
- 需要企业权限、审批、结算和用量管理的商业客户。

这些场景未来可以支持，但不进入 MVP-0。

## 5. 用户核心任务

用户真正要完成的不是“和多个 Agent 聊天”，而是：

1. 我有一个复杂问题，但不知道该找哪些专家。
2. 我希望不同专家互相质疑，而不是各自给一段建议。
3. 我希望看到哪些结论可靠，哪些只是猜测。
4. 我希望知道最终方案为什么这样定。
5. 我希望下次遇到类似问题时，可以复用这套工作流。
6. 我希望可以增加新的专业员工，并让系统按规则使用它。
7. 我希望可以直接提出诉求，让系统先匹配已有 Agent；不够时再生成个人 Agent，并允许后续共享。
8. 我希望未来可以把这个能力放到云上，给部门同事一起用。

## 6. 体验原则

### 6.1 结论必须可追溯

最终报告中的重要结论必须能追溯到：

- 哪些 Agent 参与。
- Agent 的具体版本。
- Agent 基于哪些输入材料。
- 是否被其他 Agent 质疑。
- 质疑后是否修正。
- 采纳理由是什么。
- 哪些证据仍然缺失。

### 6.2 圆桌必须有冲突

没有质疑的多 Agent 输出没有价值。圆桌至少要包含：

- 首轮独立观点。
- 交叉质疑。
- 回应与反驳。
- 修正后的观点。
- 共识与剩余分歧。

### 6.3 用户不应被迫手工调度

用户可以调整 Agent，但系统必须先给出专业推荐。

推荐必须解释：

- 为什么选择这个 Agent。
- 这个 Agent 覆盖了什么问题维度。
- 哪些候选 Agent 被排除。
- 排除原因是什么。
- 当前组合仍然缺少什么视角。

### 6.4 新增 Agent 默认不可信

新增员工不能直接进入默认推荐池。必须经过：

- 格式校验。
- 能力声明解析。
- 边界检查。
- 样例任务试运行。
- 人工确认或系统验收。
- 信任状态更新。
- 个人 Agent 默认只归属创建者，未共享前不进入公共推荐池。

### 6.5 讨论过程必须可视化

圆桌现场不应只展示聊天流。用户需要同时看到：

- 当前讨论阶段。
- 每个 Agent 的立场。
- 主要争议点。
- 哪些观点被质疑。
- 哪些观点已经修正。
- 哪些证据缺失。
- 最终共识如何形成。

### 6.6 默认保护用户材料

用户材料默认不进入长期保存。保存、导出、复用、给 Agent 访问完整材料，都应有明确用户动作或清晰标识。

## 7. 体验红线

- 禁止把产品做成多 Agent 群聊。
- 禁止只按 Agent 名称或描述检索专家。
- 禁止让新增 Agent 直接进入默认推荐池。
- 禁止最终报告只总结聊天内容而不绑定证据和质疑链。
- 禁止用绿色成功态掩盖部分失败、跳过阶段或未完成质疑。
- 禁止默认全量保存用户上传材料。
- 禁止让工具、文件、网络权限只靠 Prompt 约束，必须有系统层策略。
- 禁止隐藏“当前结论基于哪些未确认前提”。

## 8. MVP-0 成功标准

MVP-0 被认为成立，必须满足：

- 用户能在 10 分钟内完成一次完整圆桌。
- 用户无需手工挑选 Agent，也能得到合理专家组合。
- 用户能看懂为什么这些 Agent 被选中。
- 用户能看到至少 3 个明确争议点或风险点。
- 最终报告能标出结论、证据、分歧、风险和待确认前提。
- 用户能从诉求直接进入匹配或生成个人 Agent 的流程。
- 新增 Agent 能通过界面完成导入、测试和试运行状态管理。
- 一次圆桌可以保存成可复用模板。

## 9. 当前假设

- 首版以本地个人工作台为主，不优先做团队协作；云端给部门同事使用属于后续版本。
- 首版读取本机 `.codex/agents/*.toml` 和 `agency-agents` 源文件。
- 首版优先 Markdown、JSON、可视化页面，不优先 PPT/Word 导出。
- 首版以结构化事件为核心数据，不以聊天消息为核心数据。
- 首版用户可以暂停、补充材料、要求反驳、提前综合。

## 10. 待确认前提

- open spec 最终是否迁移到正式 spec-kit 仓库。
- 首版是否使用现有 Codex Agent TOML 作为唯一 Agent 源。
- 首版是否需要直接调用真实 subagent，还是先模拟执行协议。
- 首版是否必须支持联网证据核验。
- 首版是否要做桌面应用、Web 本地应用，还是 Codex 插件式界面。
- 云端版本的 AI 运行时按 OpenAI API 设计，Codex 不作为用户侧运行时。

## 11. 未来版本方向

- 云端部署给部门同事使用。
- 云端 AI 运行时使用 OpenAI API、Responses API 和必要时的 Agents SDK。
- Codex 用于产品开发、代码维护、代码审查和自动化工程任务。
- 共享范围从当前部署空间扩展到部门空间。
- 增加组织级权限、审计、同步和隔离。
- 让个人 Agent 可以在部门内受控共享。
- 让部门内的圆桌结果可以沉淀为部门知识资产。
